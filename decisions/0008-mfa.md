# 0008 — Multi-factor authentication

**Status:** Accepted
**Date:** 2026-04-25

## Context

[ADR 0004](./0004-identity-model.md) deferred MFA explicitly. The v0.1 deferral was the right call — the credential model needed to settle before MFA could be designed on top of it, and shipping `password + passkey + OIDC` as separate credential types built the right shape (a user has N credentials of different types) for MFA to inherit.

Mid-2026, the constraints have changed. Phishing-resistant MFA is no longer optional for any application that touches money, healthcare, or PII. Adopters evaluating Flametrench against rolling their own identity layer expect first-class MFA in the box. v0.1 punted this to "use multiple credentials per user and check N before issuing a session" — accurate but operationally insufficient: it leaves enrollment, recovery, grace windows, and assertion verification as the host's problem.

This ADR designs the v0.2 multi-factor authentication surface: which factors are first-class, how enrollment and verification flow, how the credential model from ADR 0004 extends, and what the cross-SDK parity contract is.

## Decision

### Scope: TOTP + WebAuthn + recovery codes

Three factor types are first-class in v0.2:

- **TOTP** (RFC 6238). The widest baseline. Authenticator apps (Google Authenticator, 1Password, Authy, Yubico Authenticator). 30-second window, 6-digit codes, HMAC-SHA1 default with HMAC-SHA256 permitted.
- **WebAuthn assertion verification** — full FIDO2 / passkey assertion. v0.1 stored passkey credentials but punted the assertion verification to the host application; v0.2 owns it end-to-end, including signature counter monotonicity checks, RP ID validation, and user-verified flag enforcement.
- **Recovery codes** — 10 single-use codes, generated at enrollment, stored only as Argon2id hashes server-side. The fallback when a user loses their primary factor.

Explicitly deferred for v0.3+: SMS (carrier insecurity is well-documented; PCI DSS 4.0 already deprecates SMS-based MFA), email magic-link as a *factor* (acceptable as account recovery; not as a daily-driver factor), and step-up by recent-auth-window (a session-design question that overlaps with v0.2 session work).

### Factors are not credentials

The v0.1 credential model has a discriminated union over `password | passkey | oidc`. v0.2 introduces a parallel concept: a **factor** is something the user proves possession of *in addition to* their primary credential. The two models are distinct because:

1. A passkey can be either a primary credential (replaces the password) or an additional factor (supplements the password). The user shouldn't have to enroll the same passkey twice.
2. Factors have their own lifecycle (suspend, revoke, re-enroll) independent of the credentials they augment.
3. Recovery codes are not credentials; they are factors that exist only to bypass the requirement to present another factor.

A factor is stored as a new entity type, `mfa_`, with type prefix added to the registry. The `mfa_` shape:

```
mfa_<hex>
  - id: mfa_<32hex>
  - usr_id: ref usr_
  - type: 'totp' | 'webauthn' | 'recovery'
  - status: 'active' | 'suspended' | 'revoked'
  - identifier: type-specific (label for totp; credential id for webauthn; null for recovery)
  - replaces: nullable mfa_id (for re-enrollment chain)
  - created_at, updated_at
  - <type-specific payload, internal only>
```

Type-specific payload (never exposed publicly, parallel to how v0.1 hides `password_hash`):

- **TOTP**: shared secret (16+ bytes), algorithm (SHA1 / SHA256), digits (6 or 8), period (30s default).
- **WebAuthn**: public key bytes (COSE-encoded), signature counter, RP ID, AAGUID for authenticator metadata, transports list.
- **Recovery**: the 10 PHC-encoded Argon2id hashes; a parallel boolean array tracking which have been consumed.

Revoke-and-re-add ([ADR 0005](./0005-revoke-and-re-add.md)) applies to factor rotation: re-enrolling a TOTP factor revokes the old `mfa_` row and inserts a new one with `replaces = old.id`.

### MFA enrollment

`enrollMfa(usrId, type, ...type-specific-args)` returns:

- For TOTP: the secret (base32-encoded) and the otpauth:// URI for QR rendering. The factor is created in `status: pending` until `confirmEnrollment(mfaId, code)` validates a fresh TOTP code; confirmation flips it to `active`.
- For WebAuthn: the registration challenge and required client params. After the client returns the attestation, `confirmEnrollment(mfaId, attestation)` validates the signature and stores the public key.
- For Recovery: the 10 plaintext codes (returned ONCE; user MUST save them) and the resulting `mfa_` row in `active` status. Recovery codes have no two-step confirmation — they are useful immediately.

The pending → active transition is the spec-required confirmation step. Implementations MUST NOT mark TOTP or WebAuthn factors active until the user has demonstrated possession via one validation. This prevents the "user enrolls a factor, locks themselves out before testing it" failure mode.

Pending factors expire after 10 minutes if not confirmed. Implementations MUST NOT keep an unbounded queue of pending enrollments per user.

### Verification flow

`verifyMfa(usrId, factorChallenge)` accepts:

- TOTP: `{ type: 'totp', code: '123456' }`. The SDK enumerates the user's active TOTP factors, validates against each within a ±1 window (drift tolerance per RFC 6238), and rejects code reuse via a per-(usr_id, code, time-step) deduplication index.
- WebAuthn: `{ type: 'webauthn', assertion: <ClientData + AuthenticatorData + signature> }`. The SDK looks up the factor by credential ID, verifies the signature, validates the RP ID, asserts the user-verified flag (the `UV` bit), and increments the stored signature counter. A counter that would decrease MUST be rejected — that is the WebAuthn-spec test for cloned authenticators.
- Recovery: `{ type: 'recovery', code: 'XXXX-XXXX-XXXX' }`. The SDK iterates active recovery hashes, finds the match, marks that slot consumed, and returns success. A user with all 10 codes consumed has no recovery available; re-enrollment generates a fresh set and revokes the old.

`verifyMfa` returns a boolean and the matched `mfa_id` on success. It does NOT issue a session by itself — the session-issue path is `verifyPassword → verifyMfa → createSession`, three steps the application sequences.

### Session lifecycle changes

The session entity gains an optional `mfa_verified_at` timestamp. A session created without an MFA verification has `mfa_verified_at = null`. Applications enforcing MFA gate sensitive operations on `session.mfa_verified_at IS NOT NULL AND now() - session.mfa_verified_at < <step-up-window>`.

The spec floor for the MFA verification window is 30 minutes from `mfa_verified_at`. Applications MAY tighten this; implementations MUST honor whatever window the application passes to `createSession({ ..., requireMfaWithin: <seconds> })`.

`refreshSession` does NOT carry `mfa_verified_at` forward. A refresh that requires MFA must invoke `verifyMfa` first; this matches the security goal that MFA freshness should not be amortizable.

### Per-user MFA policy

A user's MFA enforcement policy lives in a new `usr_mfa_policy` shape:

```
usr_mfa_policy
  - usr_id: ref usr_ (PK)
  - required: boolean
  - grace_until: nullable timestamptz (a deadline before which login still works without MFA)
  - updated_at
```

When `required = true` and `grace_until` is null or past, `verifyPassword` MUST also produce a "MFA required" signal — the application then chains into `verifyMfa` before issuing the session. The signal is a structured response, not an exception, because "MFA required" is a happy-path step in the auth flow.

Org-level "all members must enable MFA" enforcement is an application concern that consumes `usr_mfa_policy` for each member; the spec does not extend the org schema for this. Apps that want it write a periodic check that updates each member's policy.

### Recovery code semantics

- Codes are 12 ASCII characters, four-character groups separated by hyphens (`XXXX-XXXX-XXXX`). Easy to read aloud, hard to mistype. Generated from `random_bytes(9)` base32-encoded; the alphabet excludes `0/O/1/I/L` to avoid ambiguity.
- Each code is single-use. The set of 10 lasts until consumed or until the user explicitly regenerates.
- Storage: each code is hashed independently with Argon2id at the spec floor (same parameters as password credentials). The `mfa_` row stores the array of hashes plus a parallel array of `consumed_at` timestamps.
- Constant-time verification: the SDK iterates all 10 hashes regardless of where the match is (or that no match exists) to avoid leaking timing about which slot matched. Implementations using `argon2.verify` natively get this for free.
- On consumption, the slot's `consumed_at` is set; the hash remains so the row stays a complete record of which codes were used.

A user with 0 unconsumed codes has lost their recovery factor. Re-enrolling generates a fresh set; the old `mfa_` row is revoked per ADR 0005.

### Cross-SDK parity contract

The conformance suite gains a `fixtures/identity/mfa/` corpus:

- **`totp-rfc6238.json`** — RFC 6238 test vectors (spec-pinned secrets, time steps, expected codes). Verifies that every SDK's TOTP implementation produces identical codes for identical inputs.
- **`totp-drift-tolerance.json`** — verifies that codes from `t-30s` and `t+30s` are accepted; codes from `t-60s` are rejected.
- **`totp-replay-rejected.json`** — verifies that the same code submitted twice within its window raises `ReplayedTotpError` on the second attempt.
- **`webauthn-assertion.json`** — synthetic assertion against a known public key; verifies signature + RP ID + UV bit.
- **`webauthn-counter-decrease-rejected.json`** — verifies cloned-authenticator detection.
- **`recovery-code-format.json`** — verifies the alphabet and grouping rules.
- **`recovery-code-consumed.json`** — verifies that a consumed code is rejected on re-presentation.
- **`mfa-required-signal.json`** — verifies that `verifyPassword` with `required = true` returns the MFA-required signal rather than minting a session.

The TOTP fixture is RFC 6238-anchored: the test vectors are from the RFC's appendix. This makes Flametrench's TOTP byte-identical to every other RFC-conformant implementation, not just to itself across SDKs.

## Consequences

**Positive:**
- MFA is in the box. Adopters stop reinventing TOTP enrollment QR codes and recovery-code flows from scratch.
- WebAuthn assertion verification is a single SDK call, not a 200-line dance against `@simplewebauthn` (Node) / `web-auth-library` (PHP) / `webauthn` (Python) / various Java libs.
- The `mfa_verified_at` field on sessions lets apps gate sensitive operations on fresh MFA without inventing a parallel session-tracking layer.
- Cross-SDK parity is enforceable: a TOTP code the Node SDK accepts is one the PHP, Python, and Java SDKs also accept (RFC 6238 test vectors anchor this).

**Negative:**
- The factor model adds a new entity type (`mfa_`) and a new prefix to the registry. Every SDK gains roughly +400 lines of code to handle the three factor types, enrollment, verification, and session integration.
- WebAuthn assertion verification is non-trivial — credential lookup, signature verification across multiple algorithms (ES256, RS256, EdDSA), counter monotonicity, AAGUID handling. The conformance fixture corpus needs many corner cases.
- `verifyPassword` returning a structured "MFA required" signal is a v0.2 API change. v0.1 callers continue to receive `VerifiedCredential` because their users have no `usr_mfa_policy.required = true` row; the change is API-additive, not breaking. SDKs MUST document it as such.
- Recovery codes have an irreducible UX cost: users must store them somewhere safe, and many won't. This isn't a Flametrench problem but it is a Flametrench-adopter problem; documentation MUST be explicit about the trade-off.

## Deferred

- **SMS** as a factor. Carrier-side SIM swap risk and PCI DSS 4.0 deprecation rule it out. Emergency-only fallback (account recovery) may use email instead.
- **Email magic-link as a factor.** Acceptable for password-reset flows (ADR 0006 covers legacy migration); not acceptable as a daily-driver factor because mailbox compromise == account compromise. Out of scope.
- **Step-up by recent-auth-window** as a separate concept. Subsumed by `mfa_verified_at` on sessions for v0.2; revisit as a first-class concept in v0.3 if adopter demand emerges.
- **Hardware token enrollment beyond WebAuthn** (e.g., Yubico OATH-TOTP enrolled directly without going through an authenticator app). WebAuthn covers the security-key case; OATH-direct is a niche we revisit if requested.
- **MFA at the org level** (e.g., "all members of org_X must enable MFA"). This is an application policy on top of `usr_mfa_policy`, not a spec-level entity. Documentation MAY include a recipe; the schema does not.
- **Per-resource MFA policies** ("require MFA in the last 5 minutes to delete a project"). Composable from `mfa_verified_at` and the application's own authz; spec does not extend.
- **Cross-device passkey synchronization concerns.** Modern passkeys roam (iCloud Keychain, Google Password Manager). The spec is neutral on whether a particular WebAuthn credential is hardware-bound or cloud-synced; the AAGUID exposes this metadata for apps that want to enforce a policy.

## Rejected alternatives

### MFA as a credential type (extend the v0.1 credential union)

Rejected. A passkey already has both roles — primary credential (replaces a password) and additional factor (supplements one). Encoding factor-vs-credential as a status flag on a single union creates type-system contortions in every SDK's `Credential` discriminated union. The two concepts are operationally distinct enough to deserve distinct entities.

### Embed factor verification in `verifyPassword`

Rejected. `verifyPassword` is a primitive: "did this plaintext produce that hash?" Bundling factor verification into it would couple the password-verify path to MFA configuration, complicate the legacy-migration story (ADR 0006), and make it harder for apps to invoke factor verification independently (e.g., for step-up before a sensitive action).

The two-step flow (`verifyPassword → verifyMfa → createSession`) is more code on the application side but cleanly separates the two security primitives. Docs MUST include a worked example.

### Use a third-party MFA library per language

Tempting because TOTP is RFC 6238 and WebAuthn libraries exist in every language. Rejected because:

- Cross-SDK parity becomes "the libraries we picked happen to agree." Bugs in one library's edge-case handling break Flametrench's parity claim.
- The conformance fixture corpus is meaningless if the SDK delegates to another library — we'd be testing the third-party library, not Flametrench.
- The total surface area is small enough (TOTP HMAC step + WebAuthn signature verification + recovery hash) that owning it is cheaper than vetting and tracking four separate library lineages.

The v0.2 implementation MAY use a vetted library internally for the cryptographic primitives (ECDSA, HMAC) — those are commodity and not Flametrench's value-add. The factor-verification *flow* (challenge generation, replay protection, counter monotonicity) is Flametrench's; we own its semantics.

### One-step enrollment (skip the pending → active transition)

Rejected. Without the confirmation step, a user can enroll a TOTP factor on a device they then lose access to (typo'd the secret into the wrong app, etc.) and immediately lock themselves out on the next required-MFA login. The cost of one extra round-trip at enrollment time is small; the cost of skipping it is a support-ticket category we don't want to create.

### Recovery codes as a separate entity type

Rejected. Recovery codes share the same lifecycle hooks as TOTP and WebAuthn factors (suspend, revoke, re-enroll), the same per-user limits (one set at a time), and the same audit story (an `mfa_` row that gets a `revoked_at`). Forcing them into a parallel entity type duplicates the lifecycle machinery without benefit.
